SEÇÃO 01 // ORIGEM

A Origem
do Caos Controlado

Em 1996, Kent Beck foi contratado pela Chrysler para resgatar o projeto C3 (Comprehensive Compensation System) — um sistema de folha de pagamento que estava falhando espetacularmente.

Diante do desastre, Beck decidiu algo radical: levar todas as boas práticas de desenvolvimento ao extremo absoluto.

Se revisar código é bom? Todos revisam o tempo todo. Se testar é bom? Testamos antes de codificar. Se integrar é bom? Integramos várias vezes ao dia.

SEÇÃO 02 // CORE VALUES

Os 5 Valores Fundamentais

Clique em cada card para revelar mais detalhes.

Comunicação

Olho no olho, feedback constante.

CLIQUE PARA DETALHES

COMUNICAÇÃO

No XP, a comunicação face-a-face é prioridade absoluta. Documentação existe, mas conversas diretas resolvem problemas mais rápido. O valor está na troca humana.

Simplicidade

Fazer apenas o necessário para hoje.

CLIQUE PARA DETALHES

SIMPLICIDADE

Não antecipe necessidades futuras. YAGNI: You Aren't Gonna Need It. Código simples é mais fácil de mudar, testar e entender.

Feedback

Ciclos curtos de entrega.

CLIQUE PARA DETALHES

FEEDBACK

Entregas frequentes = feedback frequente = direção correta. Testes, demos ao cliente, CI — tudo gera loops rápidos de feedback.

Coragem

Corrigir erros rápido. Dizer não.

CLIQUE PARA DETALHES

CORAGEM

Coragem para refatorar código ruim, dizer não a prazos inviáveis, descartar código que não funciona. Coragem disciplinada.

Respeito

Entre o time e com o cliente.

CLIQUE PARA DETALHES

RESPEITO

Respeitar a capacidade dos colegas, confiar que cada um faz o melhor. Respeitar o cliente como parceiro. Sem respeito, os outros valores desmoronam.

SEÇÃO 03 // PRÁTICAS

Ciclo de Vida e Práticas Técnicas

O XP é conhecido por ser "mão na massa". Selecione uma categoria.

Pair Programming

Dois desenvolvedores, uma máquina. O Piloto digita enquanto o Copiloto revisa em tempo real, pensa em edge cases e orienta a próxima ação.

CONCEITO
// Fluxo do Pair Programming
Piloto   → digita, foca na sintaxe
Copiloto → revisa, pensa no macro
// Troca de papéis: 15-30 min
trocarPapéis() → qualidade ++

TDD

Test Driven Development: escrever o teste antes do código. Ciclo Red-Green-Refactor.

JavaScript
// 1. RED — Teste falha
test('soma 2+3 = 5', () => {
  expect(somar(2,3)).toBe(5);
});
// 2. GREEN — Faz passar
function somar(a,b){return a+b}
// 3. REFACTOR — Melhora

Refatoração

Melhoria contínua do design sem alterar comportamento. Com TDD, refatorar é seguro.

Python
# ANTES — duplicado
def calc(val, tx):
  if val>1000:return val*tx*.9
  return val*tx
# DEPOIS — limpo
def desc(v):return .9 if v>1000 else 1.0
def calc(v,t):return v*t*desc(v)

Propriedade Coletiva

Qualquer pessoa pode modificar qualquer parte do código.

CONCEITO
// Sem prop. coletiva:
moduloA → "Só o João mexe" // ❌
// Com prop. coletiva:
tudo → "Time inteiro" // ✅
qualquerDev().podeModificar(qualquerParte)

Padrões de Codificação

O código deve parecer escrito por uma única pessoa.

.eslintrc
{
  "rules": {
    "semi": ["error","always"],
    "quotes": ["error","single"],
    "indent": ["error",2]
  }
}

Negócio (o que) e Técnico (como) negociam escopo.

SEÇÃO 04 // COMPARATIVO

XP vs Scrum

CARACTERÍSTICASCRUMEXTREME PROGRAMMING
Foco PrincipalGestão e organização do trabalhoPráticas de engenharia e qualidade de código
Mudanças na SprintNão permitidas durante a SprintPodem ocorrer a qualquer momento
PrioridadeProduct Owner defineCliente define e prioriza rigidamente
TestesRecomendadosObrigatórios — TDD é a regra
Trabalho em ParOpcionalRecomendado como prática core
IteraçãoSprint (2-4 semanas)Iterações menores (1-2 semanas)
GestãoScrum Master, PO, Time DevCoach, Cliente, Desenvolvedores

Erro Comum

Scrum gerencia o "quando" e "o que", enquanto XP define o "como" com ênfase brutal em qualidade técnica. Na prática, muitos times usam Scrum + XP — combinação poderosa.

SEÇÃO 06 // DESAFIO PRÁTICO

XP em Miniatura // 1h30

Desafio de código em duplas com hospedagem. Cada dupla deve documentar cada passo no Diário de Bordo justificando com valores e práticas do XP.

1h30 total Duplas IA permitida Diário de Bordo obrigatório

As duplas podem usar IA (ChatGPT, Copilot, etc.) para gerar código, mas devem justificar cada decisão no diário de bordo, explicando qual prática XP está sendo aplicada e por quê. O foco não é o código perfeito — é demonstrar compreensão da metodologia.

O QUE FAZER

Criar um arquivo validacoes.js (ou validacoes.py) com 3 funções:

  • ▸ ehEmailValido(str) — retorna true/false
  • ▸ ehCPFValido(str) — retorna true/false
  • ▸ ehSenhaForte(str) — min 8 chars, maiúscula, número, especial

Regra obrigatória: Para CADA função, escrever pelo menos 5 testes ANTES de implementar. Usar Jest (JS) ou unittest (Python).

XP APLICADO

Valor: Simplicidade (implementar o mínimo para passar) + Feedback (testes como feedback imediato). Prática: TDD.

CÓDIGO LEGADO A REFATORAR

ANTES — refatorar.js
function processar(d,t){
  var r=0;
  if(t=='VIP'){if(d>500){r=d*.8}else{r=d*.9}}
  else{if(d>500){r=d*.95}else{r=d}}
  if(r>1000){r=r-50}
  return r
}

function formatar(v){
  return "R$"+v.toFixed(2)
}

Regras: 1) Escrever testes que cubram TODOS os caminhos antes de tocar no código. 2) Refatorar em passos pequenos, rodando testes a cada mudança. 3) O resultado final deve ter o MESMO comportamento.

DISCUSSÃO // 10 MINUTOS

"Qual prática do XP vocês acham mais difícil de implementar no mercado hoje?"

Geralmente a resposta é Pair Programming (gestores veem dois na mesma máquina como desperdício) ou Cliente Presente (nem todo cliente pode se dedicar em tempo integral).

O XP não é apenas sobre rapidez — é sobre criar um ambiente onde a mudança não é cara e a qualidade é inegociável.
